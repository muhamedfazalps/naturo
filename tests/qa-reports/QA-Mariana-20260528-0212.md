## Phase 4 — Simulated User: Enterprise RPA Dev

Mindset: I work at a Fortune-500 RPA shop. I'm wiring naturo into an existing pipeline that automates a legacy WPF accounts-payable app. I write a 10-step workflow with error checks and JSON-RPC parsing. I expect:
- Pre-flight check that doesn't crash before step 1
- Predictable exit codes I can branch on
- JSON envelopes I can pipe to `jq` / Python parsers
- Recovery when a window blocks the flow

I wrote `rpa-workflow.sh` — a 10-step Notepad save flow with `set -u`, `trap ERR`, JSON parsing of every response, and exit-code branching. Workflow stopped at **step 2** (desktop probe) because `NO_DESKTOP_SESSION` is unavoidable in this session. But the experience surfaced real issues an RPA dev would care about:

1. **No `naturo doctor` / preflight command.** I had to invent one by parsing `list apps -j` for `NO_DESKTOP_SESSION`. An enterprise dev expects `naturo doctor` to tell them whether the runtime is healthy *before* they write 10 steps. (Feature suggestion — not filing yet.)
2. **Inconsistent error contracts (#866 + scope expansion).** The same logical error (`NO_DESKTOP_SESSION`) produces:
   - exit 1 + clean text for `see / capture / list / app launch / find / wait / get`
   - exit 2 + `Usage:` banner for `type / press / click / drag / highlight` (text mode only — `-j` is correct)
   The text path makes `set -e` scripts misclassify environment failures as argument errors.
3. **JSON contract leak under optional deps (#869).** If `naturo` is installed without `[desktop]`, `desktop list -j` mixes prose with JSON. An enterprise dev piping through `jq` gets a parse error and has no clue why — the JSON `error.code` is `MISSING_DEPENDENCY`, which is the right value, but it's hidden behind two lines of human prose.
4. **`naturo clipboard` works in this session.** Worth noting — clipboard is a viable side-channel for an RPA dev when the desktop is unavailable (e.g., reading content set by another process). It's the one input/output surface that survived this session.

## Top 3 Risks

1. **#869 — JSON contract leak under optional deps.** An AI agent or RPA script that pipes `-j` output through a parser gets a JSONDecodeError, with no signal that the cause is a TTY-detection bug. Worse, if stdout is a TTY too the script will hang on `input()`. Cheap one-line fix (check `json_output` in `_is_interactive`) but it's a class of bug that may exist in other prompt/confirm paths.
2. **#863 ship gate — 6 consecutive rounds.** v0.3.2 cannot be released until the 5 SendInput-dependent fixes are verified. Same workarounds still on the table (console session, cloud VM #860 day 22 unassigned, accept code-review + unit-test verification).
3. **#866 wider than the title.** The fix landing for `type/press/click` must also cover `drag` and `highlight` (same `_get_backend` codepath). If the fix is scoped narrowly, the asymmetry will persist for two more commands and break the same scripts.
