**Where naturo wins for the evaluator**:
- Imports cleanly on a stock Python 3.12 install; pywinauto needs admin or per-user venv to work around comtypes' write-to-site-packages assumption.
- MCP surface (64 tools) is the killer differentiator — pywinauto has 0, pyautogui has 0. An AI agent author trying to add Windows desktop control to Claude/GPT will pick naturo on this alone.
- `--app` filter + `eN` element refs + `-j` envelope: pywinauto requires writing Python; naturo offers a single CLI shot.

**Where naturo loses for the evaluator** (newly noticed under this persona):

1. **`app windows -j` silently succeeds in NO_DESKTOP** while `see / capture / list windows / list apps` all refuse — filed as **#878 (P1, silent-failure)**. An AI agent calling `app windows` first to discover hwnds will get a list of unusable handles and burn its turn budget on follow-up failures.
2. **`browser launch -j` envelope drift** — returns `{"status":"ok", "pid":...}` instead of `{"success":true,...}`. Every other browser subcommand (tabs/title/url/...) uses the standard envelope. Filed as **#879 (P2)**.
3. **No `naturo browser quit`** — `browser launch --headless -j` spawned 9 Chrome processes I had to clean up via PowerShell. Not filing yet; want to confirm it's not just hidden under a different name.
4. **Chinese mojibake in stderr** — the embedded WinError 10061 message inside the browser-connection error came through as `由于...` (Chinese-locale OS error). Not filed — system locale issue, partially covered by #871.
