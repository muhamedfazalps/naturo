## Phase 2 / 3 — Read-Only Accessibility Surface Probes

Cannot run E2E from NO_DESKTOP_SESSION. Probed naturo's accessibility surface via help text, source schema, and `-j` envelope inspection.

### `UIElement` schema (read from `naturo/models/snapshot.py:16-49`)

Current fields: `id, elementId, role, title, label, value, description, identifier, frame, isActionable, parentId, children, keyboardShortcut`

**Missing for keyboard-only / screen-reader-aware automation:**

| UIA property | Used today? | Impact when missing |
|--------------|-------------|---------------------|
| `IsKeyboardFocusable` | **NO** (zero references in `naturo/` or `core/`) | Cannot answer "can Tab reach this?" |
| `HasKeyboardFocus` | **NO** | Cannot query which element currently has focus |
| `HelpText` | **NO** | Tooltip / extended description not exposed (screen-reader gap) |
| `IsEnabled` | **NO** (only `isActionable`, which conflates) | Disabled-but-actionable elements indistinguishable |
| `IsOffscreen` | **NO** | Scrolled-off elements look clickable but click misses |
| `LocalizedControlType` | **NO** | Non-English locales get English roles only — Narrator says "按钮", naturo says "Button" |
| `AcceleratorKey`/`AccessKey` (→ `keyboardShortcut`) | Field exists, **never populated** (#886) | Already filed |

`grep -rnE 'IsKeyboardFocusable|HasKeyboardFocus|HelpText|IsEnabled|IsOffscreen|LocalizedControlType' naturo/ core/` returns zero hits in source. The only repo-wide match is in a prior QA report. Filed as **#896** (P2 v0.3.4) — distinct from #886 (data-population) since this is schema-coverage.

### `naturo find` filter surface

```
--role TEXT       Filter by element role
--actionable      Only show actionable elements
```

No `--focusable` / `--enabled` / `--visible`. A keyboard-only agent cannot ask "list Tab-reachable enabled visible elements" — must enumerate all and post-filter on properties that don't exist.

### `naturo press --on` / `naturo type --on` focus mechanism

Help text for both: *"target element — **click to focus** before pressing/typing"*.

Internal `focus_element_uia` (`naturo/backends/windows/_input.py:601`) exists and calls `IUIAutomationElement::SetFocus()` directly — takes `hwnd=`, `name=`, `role=`. **No CLI command surfaces this path.** A user on a mouse-restricted profile (the literal Accessibility persona) cannot reach this via the CLI — every focus operation is mouse-mediated, which #863 makes impossible from non-console sessions and which DPI/window-move breaks generally.

Captured in #896 "Suggested Fix" section as a follow-up ergonomic flag (`--focus-via uia` or `naturo focus eN`) — not a separate issue to avoid over-fragmentation.

### Other CLI ergonomic gaps observed (not filed — read-only)

- `naturo scroll` accepts both `--on TEXT` and `--id TEXT` for element targeting; every other element-targeting command (#864) accepts only `--on`. Already covered by #864 ("only `click` accepts `--id eN`") — inverse direction.
- `naturo highlight` exposes element refs via three flags: positional refs, `-r/--ref`, and `--on`. Three ways to do the same thing — inconsistent with rest of CLI which has one. Noted; not filed (cosmetic, low impact).
