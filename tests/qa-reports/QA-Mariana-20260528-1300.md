## Phase 4 — Accessibility User Persona Simulation

### What an accessibility user actually needs from `naturo see`
Tab-only navigation, screen-reader announcements, and Alt-mnemonic discovery all hinge on the element tree exposing:
- **AcceleratorKey** (e.g. `Ctrl+S` on Save)
- **AccessKey** (the Alt-prefixed access mnemonic, e.g. `Alt+F` on the File menu)
- IsKeyboardFocusable, HasKeyboardFocus, tab order, LocalizedControlType, HelpText

`naturo see` element schema today: `id, elementId, role, title, label, value, description, identifier, frame, isActionable, parentId, children, keyboardShortcut`. Of the accessibility-critical properties above, exactly **one** is in the schema (`keyboardShortcut`) and the rest are missing.

### Finding: `keyboardShortcut` is **always null** for UIA-backed elements
Empirical audit of the local snapshot cache (`~/.naturo/snapshots/`):

| Metric | Value |
|--------|-------|
| Snapshot directories | 2,835 |
| Total element records | 87,196 |
| Menu* / Button elements | 37,375 |
| **Elements with `keyboardShortcut != null`** | **0** |
| Newest 50 snapshots (today) | 49 null, 0 populated |

Read-only source dive confirms why:
- `core/src/element.cpp` (UIA backend) — zero references to `AcceleratorKey`, `AccessKey`, or `shortcut`. The UIA backend never queries or emits it.
- `core/src/msaa.cpp` and `core/src/ia2.cpp` — both *do* emit `keyboard_shortcut` (line 246 + 401). MSAA & IA2 paths work; only UIA (the default for every standard Windows app) is silently dropping the data.
- `naturo/backends/windows/_element.py:1124` docstring promises "*Fills parent_id, children IDs, and keyboard_shortcut for all elements via Python-layer post-processing.*" — but no code in that file (or any importer) actually assigns `keyboard_shortcut`; only `parent_id` and `children` are filled. The Python layer reads `props.get("keyboard_shortcut")` from C++ output, which the UIA backend never provides → always `None`.
- `naturo/backends/windows/_shell/_menu.py:244` UIA-fallback menu inspector also reads `el.keyboard_shortcut`, so `naturo menu-inspect` on UWP/Electron/WPF-custom-menu apps reports every shortcut as null too — even though the docstring claims it "displays all menu items with their keyboard shortcuts."

**Filed: #886** (P1, v0.3.4, `bug,P1,from:qa`) with the full evidence, root-cause walk, and a suggested fix (add `UIA_AcceleratorKeyPropertyId` + `UIA_AccessKeyPropertyId` queries in `element.cpp` mirroring the MSAA/IA2 emission pattern).

### Why P1 not P2
Accessibility metadata silently absent from the public contract is more than cosmetic: AI agents using `see → find → press` for keyboard-only automation get no signal for accelerator routes, so they fall back to mouse coords (brittle under DPI / window moves). Same class of issue as the silent-failure cluster #885 just consolidated — the field looks supported, but the value never lands.
